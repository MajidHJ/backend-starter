* Dependency Injection is not about avoiding singletons, but about **controlling access and coupling**
* Importing shared objects directly is a form of **architectural hardcoding**
* A dependency provider (`get_settings`) acts as a **boundary**, even if it returns a singleton
* `create_app` is a **bootstrap/composition root**, so direct access to settings there is acceptable
* Route-level dependencies must declare *what they need*, not *where it comes from*
* Proper type hints on dependencies are essential for correctness and IDE support
* `deps.py` should expose **providers only**, not core types or models
